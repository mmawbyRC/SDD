Documentation

Definitions

Internal and External Documentation

Internal Documentation - includes; comments included in the program, use of meaningful identifiers, and consistent and proper use of indentation and spacing that helps the readability as well as conveying the structure and meaning of the code.

Intrinsic Documentation - includes any content that is crucial to the program that can communicate purpose, method, intent etc.

  • Comments can be used to communicate:
    • Intent of program/function
    • Authorship of program and crediting code copied where due
  • Meaningful identifers should convey the meaning/purpose of what they name:
    • Variable names should convey the intended meaning of the variable
    • Function names should be verbs that indicate the action performed by the function
  • Indentation and Spacing
    • Align all statements that appear at the same level. Use tab to be consistent across levels
    • Use blank lines and white space to break up modules within your code to increase readability

External Documentation - seperate documents from the program; it usually consists of a user guide and sometimes includes a detailed description of the design and implementation features of the program.

System Modelling Tools

  • Context Diagram
    • DEFINITION - DFD Level 0 used to show all the external inputs and outputs to and from the system.
    • GOOD FOR -
      • Number of external entities and nature of data are good indicators of complexity of the system which informs choice of S/W development approach.
      • It can be used to define the Scope (Includes/Excludes) of a system.
    • DISTINGUISHING FEATURES -
      • Only 1 circle which represents the system
      • Data needs to be nouns, no verbs as labels on data flow arrows
      • No I/O between entities, must have a process in between entities
